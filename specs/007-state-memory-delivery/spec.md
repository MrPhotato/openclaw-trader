# 功能规格说明：状态、记忆与交付层

> **迁移说明（2026-03-15）**：`memory_assets`、`notification_service`、`replay_frontend` 和 `Macro & Event Analyst` 的主规格现位于 `specs/modules/` 与 `specs/agents/`。本文件继续保留为横切的交付与读模型 feature 文档。

**功能分支**：`codex/007-state-memory-delivery`  
**创建日期**：2026-03-11  
**状态**：草案  
**输入描述**：定义状态存储边界、记忆管理、通知交付契约、回放与前端读模型，作为模块化系统的交付与可观测性收尾层。

## 1. 背景与目标

`001-006` 已经把蓝图、公共协议、控制平面、市场与风险输出、策略执行主脊梁、Agent 协作层都定义完成。`007` 的任务是把这些结果如何落状态、形成记忆、转成通知和供前端/回放消费，完整收尾。

本 feature 覆盖：

- 状态与记忆管理模块
- 通知服务模块
- 回放与前端模块

## 1.1 2026-03-14 Macro 事件记忆收敛决议

围绕 `Macro & Event Analyst` 的记忆管理，本 feature 已确认：

- `memory_assets` 是 `Macro & Event Analyst` 事件记忆的唯一真相源。
- `memory_assets` 负责监听 `Macro & Event Analyst` 的结构化事件输出，并维护语义归并后的事件对象。
- 系统不再设计独立于 `memory_assets` 的 `Macro & Event Analyst` 私有记忆文件。
- `memory_assets` 只保存语义归并后的事件对象和按 `UTC 00:00` 切分的日记忆摘要，不保存原始新闻。
- PM 与 Risk Trader 后续读取的是 `memory_assets` 中的 Macro 事件记忆，而不是原始新闻流。
- OpenClaw 原生记忆和语义搜索后续只作为 `memory_assets` 的检索投影层，不作为新的记忆真相源。
- 若启用 OpenClaw 原生语义记忆，默认要求 `autoRecall` 开启、`autoCapture` 关闭，避免 Agent 自行写入长期事件记忆。

## 1.2 2026-03-15 真实资产收口决议

围绕多 Agent 的正式产出与系统真相，本 feature 进一步确认：

- 除 `learning` 外，系统内所有真实资产都统一由 `memory_assets` 书写和管理。
- `memory_assets` 的书写前提是：对应 Agent 已完成思考并通过对应模块提交正式决定，或其他模块已产出结构化的非 Agent 事实。
- Agent 间直接沟通、OpenClaw transcript 和协作痕迹都不是系统真相，只是协作过程。
- Agent 的决定只有在被对应模块接收、结构化并写入 `memory_assets` 后，才算系统内有效结果。
- 角色特定的 JSON 或结构化输出约束只作用于正式提交通道，不作用于 Agent 间直接沟通。

## 2. 当前系统基线

- 当前状态分散在 SQLite、reports、jsonl、logs 和 OpenClaw transcript。
- 当前通知既有确定性文案，也有 LLM 生成和 owner 转发混合逻辑。
- 当前回放素材存在，但没有统一读模型和前端查询契约。
- 本 feature 依赖 `002-006` 的所有核心契约，但不反向定义业务算法。

## 3. 用户场景与验收

### 场景 1：实现者需要统一状态与记忆边界

实现者在搭状态层时，需要知道什么是状态真相、什么是记忆提炼、什么只是事件日志。

**验收标准**

1. 必须定义状态快照、记忆视图和读模型的边界。
2. 必须明确哪些实体来自事件流聚合，哪些来自持久状态。

### 场景 2：实现者需要统一通知与回放读模型

实现者在做通知和前端时，不需要再猜测应该读 SQLite、brief 还是 transcript。

**验收标准**

1. 必须定义通知请求/通知结果契约。
2. 必须定义回放查询和前端消费模型。

## 4. 功能需求

- **FR-001**：系统必须定义 `StateSnapshot`、`MemoryView` 和至少一个回放读模型实体，明确三者边界。
- **FR-002**：系统必须定义 `NotificationCommand` 和 `NotificationResult` 契约，覆盖确定性消息、投递结果和失败原因。
- **FR-003**：系统必须定义面向前端的回放查询模型，支持按 `trace_id`、时间窗口、模块和策略版本查询。
- **FR-004**：系统必须明确状态与记忆层只消费结构化事件和核心实体，不直接依赖散装 prompt 文本作为唯一真相源。
- **FR-005**：系统必须明确前端只消费统一读模型和事件流，不直接读取零散日志文件作为主接口。
- **FR-006**：本 feature 不要求完整复刻旧系统全部状态归档细节和旧通知实现细枝末节，只定义新架构第一阶段所需能力。
- **FR-007**：系统必须支持 `Macro & Event Analyst` 的事件记忆模型，至少包含语义归并事件表和按 `UTC 00:00` 切分的日记忆摘要。
- **FR-008**：系统必须明确 `Macro & Event Analyst` 的原始新闻不进入长期记忆真相源；长期保留的只有结构化事件对象与来源引用。
- **FR-009**：系统必须支持其他 Agent 基于 `memory_assets` 主动读取 `Macro & Event Analyst` 事件记忆，而不是依赖 `Macro & Event Analyst` 主动分发普通事件。
- **FR-010**：系统必须支持将 `Macro & Event Analyst` 的结构化事件记忆投影到 OpenClaw 原生记忆搜索层，供 Agent 做关键词或语义召回。
- **FR-011**：系统必须明确 OpenClaw 原生记忆搜索层只读 `memory_assets` 投影内容，不得成为长期事件记忆的独立写入方。
- **FR-012**：除 `learning` 外，系统内所有真实资产都必须由 `memory_assets` 持久化和管理。
- **FR-013**：Agent 间直接沟通和 OpenClaw transcript 不得直接创建、修改或删除系统真实资产。
- **FR-014**：Agent 的正式决定只有在被对应模块接收并写入 `memory_assets` 后才算系统内有效结果；非 Agent 结构化事实同样遵循该收口路径。

## 5. 非功能要求

- **NFR-001**：状态、记忆、通知和回放的命名必须稳定，可供实现、前端和运维共同使用。
- **NFR-002**：读模型必须与 `002-006` 的实体兼容，不能重新发明字段名。
- **NFR-003**：通知和回放必须能够基于结构化事件重建，不依赖人工口述流程。
- **NFR-004**：事件记忆必须支持语义归并后的更新、结转和解决状态，避免同一事件因多来源报道而在记忆层重复膨胀。
- **NFR-005**：记忆真相源与语义检索索引必须保持单向同步，禁止双写、双删或双重过期逻辑。
- **NFR-006**：协作痕迹与系统真相资产必须可明确区分，避免 transcript 漂移成事实真相。

## 6. 关键实体

- **StateSnapshot**：系统当前状态快照。
- **MemoryView**：供 Agent 和复盘消费的记忆视图。
- **MemoryProjection**：由 `memory_assets` 生成并同步给 OpenClaw 原生记忆搜索的只读投影视图。
- **NotificationCommand**：通知请求。
- **NotificationResult**：通知投递结果。
- **ReplayQueryView**：前端与回放查询视图。

## 7. 假设与约束

- 本 feature 只写第一阶段所需状态与交付能力，不详细复刻旧维护归档策略。
- 前端和通知都建立在结构化事件流之上，不再以零散文本日志为主。
- 旧系统里的 `reports/*.md|json` 和 transcript 将在实现阶段被映射为过渡读模型来源，而不是目标真相源。

## 8. 成功标准

- **SC-001**：实现者可以基于 `007` 单独实现状态读模型、通知服务和回放查询层。
- **SC-002**：前端不需要直接读零散文件，就能知道应该订阅哪些事件和查询哪些读模型。
- **SC-003**：状态、记忆、通知和回放都能明确追溯到 `002-006` 的结构化契约，而不是散装经验。
