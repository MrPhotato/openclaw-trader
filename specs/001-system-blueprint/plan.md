# 实施计划：系统模块化蓝图与现状全拆解

**功能分支**：`codex/001-system-blueprint`
**规格文档**：`specs/001-system-blueprint/spec.md`
**计划日期**：2026-03-11

## 1. 执行摘要

本计划聚焦于“先把系统讲清楚，再开始重构”。本轮不直接拆代码，而是先输出一套足够细、可追踪、可执行的详细蓝图，作为后续模块化重构的统一设计基础。蓝图需要同时回答三个问题：

1. 当前系统已经做了什么、真相源在哪里、哪些代码与运行态构成现状。
2. 未来系统按 10 模块、3 条跨域平面、4 个 Agent 应该如何收口边界。
3. 如何从当前补丁式系统平稳迁移到结构化事件驱动、可回放、可前端化、可多智能体协作的新架构。
4. 如何在不泄露私密运行参数的前提下，复现当前 live 系统的关键行为边界。

## 2. 技术背景（Technical Context）

- **现有系统事实**：当前仓库基于 Python、FastAPI、Typer、Pydantic 和 SQLite；可变运行态位于 `~/.openclaw-trader/`；live 主路径为 dispatcher + perps runtime + market-intelligence + Coinbase INTX + OpenClaw。
- **目标边界**：顶层业务模块为 10 个；结构化事件、进程内事件总线和参数治理作为跨域平面；状态机与编排器持有统一外部主动触发接口；OpenClaw 被收口为多智能体协作网关的一个外部适配器。
- **主要依赖**：现有代码目录 `src/openclaw_trader/`、文档目录 `docs/`、运行时约定 `~/.openclaw-trader/`、OpenClaw 命令行调用方式、WeCom/owner 通知路由。
- **未知项 / 待确认项**：无阻塞未知项；实现期仍需根据未来前端和部署选型决定事件存储是否继续以 SQLite 为主，还是升级为独立事件存储。

## 3. 宪法检查（Constitution Check）

- 已满足“模块边界先于实现”：本轮先文档化，不直接推翻代码结构。
- 已满足“单一真相源与显式状态机”：文档将显式定义真相源与状态迁移。
- 已满足“LLM 受约束自治”：蓝图将把 LLM 和多智能体放在受约束协作层，不把核心规则下放给自然语言。
- 已满足“事件驱动与结构化可观测性”：蓝图将强制定义事件 envelope、进程内事件总线和回放链路。
- 已满足“安全渐进式迁移”：计划明确分波次实施，保留当前 live 路径。

## 4. 第 0 阶段：研究与现状归档

本阶段产出：

- 当前系统基线文档
- 当前代码与 10 模块映射表
- 当前外部接口盘点
- 当前痛点与补丁化现象归档
- 当前 live 运行态、量化模型、OpenClaw 工作区、部署拓扑与数据载体的复现手册

执行方式：

1. 读取仓库现有文档与入口代码，确认当前生产主路径与真相源顺序。
2. 用目录、入口函数、核心模型与 dispatcher/perps/runtime/strategy/state 文件建立“现状功能地图”。
3. 明确哪些能力已经存在、哪些只是未来规划、哪些目前只有半成品或兼容层。

## 5. 第 1 阶段：设计与契约

本阶段产出：

- 10 模块目标架构说明
- 3 条跨域平面定义
- 4 个 Agent 的职责与升级规则
- LLM 信息源体系与上下文视图
- 统一状态机
- 事件类型与交付协议
- 统一对外触发接口与 OpenClaw 集成契约
- 日志、回放与前端模型

设计决策：

- 继续保留 Python 主体实现，避免在架构阶段引入不必要的语言迁移。
- 继续保留 FastAPI + Typer 作为当前外部操作面，但在目标架构中统一收口成状态机与编排器持有的控制入口。
- 使用结构化事件、`memory_assets` 和进程内事件总线作为模块间协作骨架，避免多模块直接互调和状态不可追踪。
- 使用结构化事件而不是原始文本日志作为前端、回放与审计的核心数据面。

## 6. 第 2 阶段：任务分解与迁移路径

迁移会按四个波次推进：

1. **骨架波次**：先落地统一事件协议、进程内事件总线、状态机与参数治理。
2. **收口波次**：把当前代码按 10 模块逐步收口，先做边界层和适配层，不急于改深层实现。
3. **智能体波次**：把单个 `crypto-chief` 运行习惯拆成 4 个 Agent，并迁移到多智能体协作网关。
4. **可视化波次**：接入回放、前端、通知和人工调参界面。
