# 功能规格说明：系统模块化蓝图与现状全拆解

**功能分支**：`codex/001-system-blueprint`  
**创建日期**：2026-03-11  
**状态**：已成稿  
**输入描述**：将当前 `openclaw-trader` 系统完整拆解为 10 个顶层模块、3 条跨域平面和 4 个 Agent 协作体系，形成可供重构直接使用的中文蓝图文档。

## 1. 背景与目标

当前 `openclaw-trader` 已经具备运行真实交易 runtime、市场智能、dispatcher、OpenClaw 集成、通知、报表与若干维护能力，但系统边界更多体现在代码习惯与运行经验里，而不是体现在明确的模块契约、状态机和接口文档里。随着策略逻辑、执行判断链、dispatcher 韧性与 agent 行为不断补丁化，系统已经进入“还能跑，但越来越依赖局部修补”的阶段。

本功能的目标不是直接改写所有实现，而是先把当前系统的全部能力、限制、策略经验和模块职责完整归档，形成一套可以指导后续重构的详细蓝图。蓝图必须同时覆盖：

- 当前系统事实与功能拆解
- 目标 10 模块架构
- 3 条跨域平面
- 4 个 Agent 的职责边界
- LLM 信息源体系
- 状态机与手动触发入口
- 事件协议与进程内事件总线模型
- 结构化日志、回放与前端要求
- 对外接口与 OpenClaw 集成契约
- 可执行的迁移波次与任务分解

## 2. 当前系统基线

当前系统具有以下已确认事实：

- 仓库本体是交易 runtime 与 dispatcher，不是完整 OpenClaw。
- 实际 live 真相源主要位于 `~/.openclaw-trader/` 的本地运行态目录，而不是 git 仓库。
- 活跃执行路径是 `OpenClaw / WeCom -> crypto-chief -> dispatcher -> perps runtime -> market-intelligence -> Coinbase INTX`。
- 当前主要交易对象为 `BTC / ETH / SOL` 永续；系统同时保留部分 spot 和兼容路径。
- 当前系统的核心能力已覆盖：新闻同步、市场智能、多时域量化判断、风险约束、策略生成、执行判断、执行、通知、brief/report、SQLite 状态存储。
- 当前系统的关键结构问题是：编排逻辑、运行时决策、LLM 集成、状态记录和可观测性还没有被彻底拉开边界。

## 3. 用户场景与验收

### 场景 1：架构所有者梳理当前系统全貌

作为系统所有者，我需要一套中文、可追溯的蓝图，能够一次性看清当前系统有哪些能力、这些能力散落在哪些代码和运行态里、哪些地方是稳定边界、哪些地方是临时耦合，以便后续重构不丢功能。

**验收标准**

1. 文档必须把当前仓库能力、运行态真相源、OpenClaw 集成方式、风控与执行路径完整拆出来。
2. 文档必须能把现有代码和未来 10 个模块建立清晰映射关系。

### 场景 2：重构实现者按模块推进迁移

作为后续重构实现者，我需要明确知道每个模块负责什么、输入输出是什么、哪些数据是它的真相源、它如何通过结构化事件、`memory_assets` 和进程内事件总线与其他模块协作、状态机如何切换、手动触发接口在哪里，以便按波次迁移而不是继续边修边散。

**验收标准**

1. 文档必须提供 10 模块职责、非职责、接口契约、事件模型、状态机和迁移顺序。
2. 文档必须明确 4 个 Agent 的职责边界、自主权限、升级路径和与 Crypto Chief 的关系。

### 场景 3：前端与运营构建实时可视化和回放

作为前端或运营使用者，我需要系统输出统一的结构化事件和日志，使实时监控、历史回放、参数调节、通知呈现和异常定位都能基于同一套协议工作。

**验收标准**

1. 文档必须定义结构化事件协议、事件类型规划、回放链路和前端关注视图。
2. 文档必须说明哪些参数可以手动调整、如何审计、如何通过统一外部入口触发系统动作。

## 4. 功能需求

- **FR-001**：必须形成一份当前系统基线文档，完整描述现有运行路径、真相源、关键模块、代码映射、主要痛点和当前保留的兼容路径。
- **FR-002**：必须定义 10 个顶层业务模块，并为每个模块给出职责、输入、输出、边界、不负责事项和当前代码落点。
- **FR-003**：必须定义 3 条跨域平面：结构化事件与日志平面、事件协议与进程内总线平面、配置与参数治理平面。
- **FR-004**：必须定义 4 个 Agent：策略经理（PM）、风险交易员、宏观与事件分析师、Crypto Chief，并明确它们的职责、输入输出、自主权限与升级路径。
- **FR-005**：必须定义 LLM 可消费的信息源分类，并说明每一类信息源归属于哪个模块、以何种视图供不同任务或 Agent 使用。
- **FR-006**：必须定义统一状态机，覆盖从数据就绪到策略生成、执行判断、执行、通知、回放和失败降级的主要状态与迁移规则。
- **FR-007**：必须定义统一外部主动触发入口，并明确该入口由状态机与编排器模块持有。
- **FR-008**：必须定义系统对外接口，包括当前 CLI/FastAPI 面、未来统一控制接口，以及与 OpenClaw 的独立集成契约。
- **FR-009**：必须定义模块间接口与事件协议，包括事件 envelope、事件类型规划、命令与回执格式。
- **FR-010**：必须定义日志、回放与前端展示所依赖的结构化事件、存储和查询模型。
- **FR-011**：必须给出从当前系统迁移到目标架构的波次计划与任务分解，确保可渐进落地。
- **FR-012**：必须补充复现级文档，覆盖当前 live 运行态配置、量化小模型训练与推理、OpenClaw/Agent 工作区、部署拓扑以及接口与数据载体，使后续团队能够在不查看大量源码的前提下重建现有行为边界。

## 5. 非功能要求

- **NFR-001**：文档默认使用中文；确需保留英文时，只用于专业术语、代码标识、消息主题或接口字段名。
- **NFR-002**：所有架构说明都必须与当前仓库和运行态事实一致，不能只写理想设计而不标注现状落点。
- **NFR-003**：所有模块、接口、状态与消息定义都必须具备可追溯编号或命名，方便前端、后端、Agent 与运维共用。
- **NFR-004**：蓝图必须支持保留现有 Python/FastAPI/Typer/Pydantic/SQLite 体系，并在此基础上沿用进程内事件总线、多智能体协作和 `memory_assets` 真相层。
- **NFR-005**：蓝图必须显式约束 LLM 的权限边界，防止未来重构把不可回放的自然语言决策重新变成核心真相。
- **NFR-006**：复现级文档必须与 10 模块架构兼容，描述当前事实时不能脱离未来模块边界；同时不得把本地私密 token、key、secret 写入仓库。

## 6. 关键实体

- **模块（Module）**：顶层业务能力单元，拥有明确职责、端口、事件和数据所有权。
- **跨域平面（Cross-Cutting Plane）**：服务所有模块的一致性基础设施层，如进程内事件总线、日志事件和参数治理。
- **信息源（Information Source）**：被 LLM、Agent、策略或前端消费的结构化事实集合，例如新闻摘要、量化判断、风险边界、账户状态。
- **上下文视图（Context View）**：为某个任务或 Agent 组装后的信息子集，例如策略视图、执行判断视图、事件分析视图。
- **工作流状态（Workflow State）**：系统在显式状态机中的某个阶段，例如 `strategy_ready`、`execution_judgment_pending`。
- **事件信封（Event Envelope）**：用于日志、回放、内存事件总线和持久化镜像的统一事件协议。
- **手动触发命令（Manual Trigger Command）**：通过统一外部入口驱动系统执行某类动作的命令对象。
- **Agent 任务（Agent Task）**：发给某个智能体的一次有边界任务请求，带上下文视图与输出契约。

## 7. 假设与约束

- 当前仓库仍需兼容现有运行态 `~/.openclaw-trader/`，不能在蓝图阶段假定所有运行数据都会立刻迁移。
- 当前 live 系统仍会依赖 OpenClaw 作为外部智能体运行环境，因此 OpenClaw 集成必须被保留为独立适配层。
- 当前系统存在若干已知补丁和过渡路径；蓝图需要把它们记下来，而不是假装它们不存在。
- 本轮交付目标是“文档级蓝图与任务设计”，不是一次性完成代码重构。

## 8. 成功标准

- **SC-001**：仓库中形成一套完整中文蓝图，能把当前系统的主要功能与未来 10 模块一一对应起来。
- **SC-002**：未来任何一个模块化重构任务，都能明确追溯到本功能文档中的模块边界、接口契约、状态机或任务项。
- **SC-003**：前端、Agent、交易执行、通知与状态回放不再依赖口头经验，可以直接基于文档对齐数据流和事件流。
- **SC-004**：量化模型、OpenClaw 工作区、部署拓扑和本地运行态不再依赖“知道的人”，而有一套与目标架构兼容的复现手册可查。
