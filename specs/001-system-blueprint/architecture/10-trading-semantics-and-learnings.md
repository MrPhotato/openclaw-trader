# 当前交易语义与经验沉淀

本文件记录“当前系统现在实际上如何做交易判断”，以及这段时间已经验证过的关键经验。

## 1. 当前交易语义

### 1.1 多时域职责

当前有效语义不是旧文档里“`1h` 决定 trigger”的版本，而是：

- `12h`：决定大方向锚点
- `4h`：决定是否开仓、是否加仓、基础仓位档位
- `1h`：只负责已有仓位的短线减仓 / 延后，不再参与新开仓判定

### 1.2 当前硬风控边界

当前本地硬边界已收敛到 `policy_risk`：

- `max_leverage = 5.0`
- `max_total_exposure_pct = 100.0`
- `max_symbol_position_pct = 66.0`
- `max_order_pct = 33.0`
- `position_risk_state = observe/reduce/exit`

LLM、执行判断和执行都必须服从它。

### 1.3 当前开仓规则

- `12h` 有明确方向且 `4h confirmed`：允许承担正常方向性风险
- `12h` 有明确方向但 `4h = flat`：可保留极小试探
- `12h neutral` 且 `4h` 有方向：只允许试探性风险
- `1h` 不再参与新开仓；若已持有与大方向一致的仓位，`1h` 可触发 `trim / wait`

### 1.4 新闻处理

新闻仍然作为上下文与事件材料进入系统，但“新鲜新闻直接拦住下单”的旧硬门槛已经移除。  
现在新闻影响应通过：

- 新闻摘要
- 事件对象
- `memory_assets` 中的事件记忆

来表达，而不是通过统一 observe gate 一刀切。

## 2. 已经确认的系统经验

### 2.1 真正的核心决策已经是量化模型 + 本地守卫

当前系统里，LLM 已经不是第一决策者。  
更准确地说，核心是：

- 量化模型
- `policy_risk`
- 风控与执行守卫
- 本地 parser / execution guard

LLM 主要负责：

- 在边界内生成策略意图
- 产出执行判断
- 生成对人友好的解释

### 2.2 `1h` 作为开仓依据价值较低

近期训练与校准结果表明，`1h` 的交易目标质量明显弱于 `4h/12h`。  
因此当前系统已经把 `1h` 降级为“已有仓位的短线风险参考”，而不是“新开仓的硬门槛”。

### 2.3 最近最有效的放宽项

以下几项已经被证明是更合理的方向：

- 允许 `12h` 有方向、`4h=flat` 时开极小试探
- 把 `1h` 从开仓链路中摘掉
- 继续把新闻保留为上下文，但不直接阻断下单
- 放宽 `4h/12h` 分类门槛与 uncertainty freeze 阈值

### 2.4 最近最重要的修复经验

以下问题已经被识别为“系统性坑”：

- 手动策略刷新吞掉例行定时刷新
- 执行判断失败后整条执行链路卡死
- 硬风控已阻断仍然生成后续执行上下文
- 执行上下文已出现但由于多余重路径没有真正触发执行判断
- Coinbase 公共行情请求失败会把整轮 dispatcher 打崩

这些经验说明：后续重构里，流程状态、执行上下文合法性、数据韧性和回执链必须先收口成明确模块。

## 3. 当前策略经验对未来蓝图的要求

### 3.1 量化判断与风控守卫必须分层

因为现在已经证明“模型输出”和“是否允许执行”是两层不同语义，所以未来不能再把它们混在一个 runtime 大文件里。

### 3.2 策略意图与执行动作必须严格分开

当前最大的实战经验之一就是：

- 策略给了目标仓位
- 不代表这分钟可以下单

未来蓝图必须保留这种分层，而不是让策略意图直接成为执行命令。

### 3.3 回放必须能解释“为什么没下单”

未来回放与前端不能只展示“有没有订单”，还必须显示：

- 哪个模块卡住了
- 当前状态机停在哪个状态
- 哪一条风险或政策阻止了迁移
- 哪个 Agent 产出了什么执行判断

## 4. 当前策略经验的归档位置

未来建议把这类经验长期归档在：

- 状态与记忆管理模块中的 `learnings`
- 回放与前端模块中的问题案例库
- 宪法与蓝图文档中仅保留稳定、长期有效的原则
