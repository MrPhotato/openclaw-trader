# LLM 信息源与上下文视图

本文件回答两个问题：

1. LLM / Agent 到底应当消费哪些信息源。
2. 不同任务应该看到哪些信息源视图，而不是看到整个系统原始世界。

## 1. 信息源分类

| 信息源类别 | 归属模块 | 说明 | 是否直接给 LLM |
| --- | --- | --- | --- |
| 市场结构化判断 | 量化判断模块 | `12h/4h/1h`、probability、regime、trade quality、诊断 | 是，优先 |
| 新闻与事件摘要 | 新闻事件模块 | 新闻摘要、事件对象、影响范围、紧急度 | 是，但只给摘要与结构化提炼 |
| 风险边界 | 风控与执行守卫模块 | `RiskGuardDecision`、风险上限、持仓风险状态、breaker | 是，必须 |
| 策略连续性 | 策略与组合意图模块 | 当前策略、前一版策略、recheck、策略变更原因 | 是 |
| 执行与账户事实 | 账户与下单模块 | 当前仓位、权益、可用保证金、预览结果、最小下单约束 | 是 |
| 状态与记忆 | 状态与记忆管理模块 | 近期策略、执行判断历史、learnings、简要轨迹 | 是，但必须筛选 |
| 任务与输出契约 | 状态机与编排器 / 多智能体协作网关 | 这一轮任务类型、输出 JSON 契约、可行动作 | 是，必须 |
| 参数与操作意图 | 配置与参数治理平面 / 状态机与编排器 | 人工触发命令、参数 override、开关状态 | 部分可见 |

## 2. 不应直接喂给 LLM 的内容

- 全量原始行情特征
- 全量订单历史
- 低层 HTTP / DNS 错误日志
- 所有 memory 原文
- 所有 reports 原文
- 多份重复快照

原则是：LLM 消费“整理后的视图”，而不是“散装真相源”。

## 3. 上下文视图

### 3.1 PM 策略视图

**消费者**：PM Agent  
**目标**：生成或更新策略意图

包含：

- 市场结构化判断
- 新闻与事件摘要
- 风险边界
- 当前策略与上版策略
- 组合与账户摘要
- 必要的近期记忆
- 输出契约

不包含：

- 原始订单执行日志
- 低层网络异常
- 无关历史 prompt transcript

### 3.2 风险交易员视图

**消费者**：Risk Trader Agent  
**目标**：执行判断、临场机会识别、执行前决策

包含：

- 当前 `ExecutionContext`
- 最新 `RiskGuardDecision`
- 风险边界与执行预览
- 当前仓位与杠杆
- 最近相关新闻与事件
- 明确的允许动作和禁止动作

### 3.3 宏观与事件分析视图

**消费者**：Macro & Event Analyst  
**目标**：整理新闻、事件、宏观背景并做升级

包含：

- 原始新闻列表
- 事件日历
- 分类标签
- 最近市场摘要
- 当前策略方向与持仓概览

### 3.4 Crypto Chief 统筹视图

**消费者**：Crypto Chief  
**目标**：和 owner 沟通、复盘、Learning、处理升级

包含：

- 策略结果摘要
- 风控结果摘要
- 下级 Agent 升级请求
- 执行与通知结果
- 核心 learnings
- 人工指令与反馈

## 4. 视图构建原则

- 视图由上下文构建模块统一生成。
- 每类视图必须有版本号与 schema。
- 每类视图都要声明“包含哪些信息源”和“明确排除哪些信息源”。
- 每类视图都要和输出契约绑定，防止 LLM 拿到不完整上下文却生成过强动作。

## 5. 与当前系统的对应关系

当前系统中，以下文件已经部分承担未来上下文视图职责：

- `strategy/inputs.py`
- `briefs.py`
- `dispatch/prompts.py`
- `strategy/__init__.py` 中的 `build_strategy_input_perps`

这些能力未来都应收口到“上下文构建模块”，而不是散落在策略、brief 和 prompt 逻辑里。
