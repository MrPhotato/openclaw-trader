# 功能规格说明：事件协议与进程内总线骨架

**功能分支**：`codex/002-event-envelope-bus`  
**创建日期**：2026-03-11  
**状态**：草案  
**输入描述**：为模块化交易平台定义统一事件信封、进程内事件总线骨架与参数治理最小模型，作为后续 `003-007` 的公共基础。

## 1. 背景与目标

`001-system-blueprint` 已经定义了 12 个顶层模块、3 条跨域平面和 4 个 Agent，但这些能力还停留在蓝图层。后续任何实现都需要先共享一套统一的事件协议、消息命名和参数治理约束，否则状态机、Agent 网关、通知、回放和前端会各自发明自己的载荷形状。

本 feature 只负责“通用协议”，不进入具体业务模块的实现细节。

## 2. 当前系统基线

- 当前系统有日志、brief、SQLite、OpenClaw transcript，但没有统一 `EventEnvelope`。
- 当前消息流主要靠本地函数调用与文件写入，尚未有 事件总线 契约。
- 当前参数生效主要依赖 `~/.openclaw-trader/config/*.yaml` 与人工修改，没有统一审计模型。
- 该 feature 对应 `001` 蓝图中的：
  - 结构化事件与日志平面
  - 事件总线平面
  - 配置与参数治理平面的最小公共模型

## 3. 用户场景与验收

### 场景 1：架构实现者需要统一事件协议

实现者要为任意模块落事件和消息时，不需要再决定字段名、事件层级、trace 规则和命名方式。

**验收标准**

1. 文档必须定义统一 `EventEnvelope` 字段、命名规则和最小约束。
2. 文档必须说明命令事件、事实事件、结果事件、失败事件的命名边界。

### 场景 2：后续 feature 需要共用事件交付规则和参数审计

`003-007` 在写自己的 contract 时，只能引用本 feature 的事件类型命名、交付语义和参数变更模型，不再自定义另一套协议。

**验收标准**

1. 文档必须定义进程内事件总线的事件类型命名规则和最小交付语义。
2. 文档必须定义参数变更记录和参数生效事件的最小 schema。

## 4. 功能需求

- **FR-001**：系统必须定义统一 `EventEnvelope`，至少包含 `event_id`、`trace_id`、`causation_id`、`module`、`entity_type`、`entity_id`、`event_type`、`event_level`、`occurred_at`、`schema_version`、`payload`、`human_summary`。
- **FR-002**：系统必须为进程内事件总线定义统一的事件类型规则，并给出至少覆盖工作流、Agent、执行、通知、回放的基础事件类型族。
- **FR-003**：系统必须定义参数治理最小模型，覆盖参数变更请求、参数变更记录、参数生效事件和回滚引用。
- **FR-004**：本 feature 必须显式说明它服务于 `003-007`，并列出后续 feature 只能复用、不应重定义的公共契约。
- **FR-005**：本 feature 不得定义任何交易业务逻辑、策略语义或 Agent 专属上下文内容。

## 5. 非功能要求

- **NFR-001**：所有字段命名、事件类型命名和 schema 命名必须稳定、可追溯、可在前端和后端共用。
- **NFR-002**：协议内容必须与具体实现语言无关，可被 Python、Node 和前端客户端共同消费。
- **NFR-003**：文档必须足够具体，使后续 feature 可以直接引用，不需要再次拍板关键协议细节。

## 6. 关键实体

- **EventEnvelope**：所有模块日志、进程内总线、回放与通知都要复用的统一事件信封。
- **ParameterChangeRecord**：记录人工或系统参数变更的审计实体。
- **ParameterEffectiveEvent**：记录参数何时在何处生效的事件实体。
- **EventRouteDescriptor**：描述事件类型、发布者、观察者职责和幂等要求的路由实体。

## 7. 假设与约束

- 不要求兼容旧 spot 路径，只围绕新的模块化系统主路径设计。
- 本 feature 只写公共协议，不写任何一个业务模块的内部算法。
- 参数治理只定义最小审计模型，不在本 feature 中定义完整 UI 或控制面板。

## 8. 成功标准

- **SC-001**：`002` 产出的事件、消息和参数契约可以被 `003-007` 直接引用，无需重命名或二次建模。
- **SC-002**：后续实现 agent 仅阅读 `002` 的 contracts，就能知道统一事件与进程内事件总线的基础规则。
- **SC-003**：`002` 不引入业务耦合；后续 feature 可以在不修改该 feature 语义的前提下展开实现。
