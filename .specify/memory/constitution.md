# OpenClaw Trader 模块化重构宪法

## 核心原则

### 1. 模块边界先于实现

系统演进必须先定义模块职责、输入、输出、所有权与不负责事项，再进入实现或重构。任何跨模块依赖都必须通过显式契约、事件或端口完成，禁止通过共享隐式状态、临时脚本或提示词约定直接耦合业务行为。

### 2. 单一真相源与显式状态机

每一类运行事实都必须有唯一真相源：参数归参数中心，运行状态归状态存储，回放事件归事件流，策略语义归策略文档与结构化记录。所有关键流程都必须可映射到显式状态机；系统不得依赖“看日志猜流程”来理解当前所处阶段。

### 3. LLM 受约束自治

LLM 与多智能体可以拥有自主性，但只能在本地确定性边界内行动。核心业务规则、风控上限、执行权限、接口校验和最终状态迁移必须由本地代码与契约控制，不得把不可回放、不可验证的自然语言推理当作唯一真相。

### 4. 事件驱动与结构化可观测性

所有模块都必须输出结构化事件与可关联日志，至少包含 `event_id`、`trace_id`、`module`、`entity_type`、`event_type`、`timestamp` 与 `payload`。前端实时动画、历史回放、审计、告警与手动诊断都必须基于统一事件协议，而不是直接消费零散文本日志。

### 5. 安全渐进式迁移

重构必须分波次推进，并保持当前 live 运行链路可理解、可回退、可验证。任何架构升级都必须先保留现有功能语义，再逐步替换内部结构；禁止一次性推翻运行态真相源、交易路径或运营入口。

## 模块化原则

- 顶层业务模块固定为 9 个：`Trade Gateway`、`news_events`、`quant_intelligence`、`policy_risk`、`workflow_orchestrator`、`agent_gateway`、`memory_assets`、`notification_service`、`replay_frontend`。
- `Trade Gateway` 是统一的交易所接入边界，内部只允许再拆成 `market_data` 与 `execution` 两个子域；它们不是并列顶层模块。
- 系统的 4 个 Agent 固定为：PM、Risk Trader、Macro & Event Analyst、Crypto Chief。
- 跨域平面固定为 3 条：结构化事件与日志平面、RabbitMQ 消息总线平面、配置与参数治理平面。
- 统一外部主动触发入口只允许收口在状态机与编排器模块。
- OpenClaw 是 Agent 协作层，不是系统真相层；Agent 间直接沟通默认允许，但沟通本身不形成系统真相。
- 除 `learning` 外，系统内所有真实资产都必须经对应模块正式接收后，由 `memory_assets` 持久化和管理。
- Agent 的正式提交统一由 `agent_gateway` 按项目内共享 schema 合同做准入校验，再通过消息总线分发给对应消费者。
- 对 OpenClaw、多智能体、交易所、前端与通知渠道的适配都必须是可替换接口，不得把第三方细节渗透到核心业务语义中。

## 治理

- **版本**：1.3.0
- **批准日期**：2026-03-11
- **最后修订**：2026-03-19
- **修订机制**：任何新增顶层模块、跨域平面、状态机主状态、外部接口入口或 LLM 自主权限边界的调整，都必须先修改本宪法与对应 blueprint 文档，再进入实现。
- **合规要求**：每次规划或重构必须检查模块边界、真相源、状态机、事件协议、外部接口、日志与回放是否与本宪法一致；不一致项必须在计划或任务中显式列出并关闭。
